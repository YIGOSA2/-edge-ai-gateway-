# Edge AI Gateway вҖ” Technical Documentation

## Overview

A multi-sensor data acquisition system built on the RK3506 embedded Linux development board. It uses a local AI model for real-time anomaly detection, combined with MQTT cloud reporting and a web dashboard, to implement a complete edge intelligent gateway.

## Hardware Platform

| Component                 | Model / Configuration              |
| ------------------------- | ---------------------------------- |
| Development Board         | RK3506EVM                          |
| Temperature & Humidity    | AHT30 (I2C, 0x38)                  |
| Accelerometer             | MPU6500 (I2C, 0x68)                |
| Development Host          | Ubuntu 22.04                       |
| Cross-compilation Toolchain | arm-buildroot-linux-gnueabihf-gcc |
| Cloud Server              | MQTT Broker (mosquitto 2.0.11)     |

## Thread Details

### 1. Sensor Thread (sensor_thread.c)

- Reads AHT30 temperature/humidity + MPU6050 3-axis acceleration every 100ms
- Obtains monotonic timestamp via `CLOCK_MONOTONIC`
- Acquires mutex lock and writes to `g_shared.data`

### 2. Storage Thread (storage_thread.c)

- Initializes SQLite database (`/root/sensor.db`)
- Reads shared data every 1 second and inserts into database
- Prints progress every 60 records

### 3. AI Inference Thread (ai_thread.c)

- Maintains a sliding window of 5 samples
- Computes acceleration variance; variance > 1,000,000 triggers vibration anomaly
- Calls `ai_predict()` to check environmental anomalies (temp > 35В°C, humi > 65%)
- Requires 3 consecutive anomaly detections before triggering alarm; result written to `g_shared.ai_status`

### 4. MQTT Thread (mqtt_thread.c)

- Reads shared data and AI status every 5 seconds
- Builds JSON message and invokes `mosquitto_pub` to publish to cloud server
- Topic: `edge-gateway/sensor`

## AI Model

### Training Pipeline

1. Collect 30 minutes of normal data + 15 minutes of anomaly data on the board (heating, shaking, breathing on sensor)
2. Export from SQLite to CSV
3. Python rule-based labeling (automatic labeling based on feature thresholds)
4. scikit-learn Random Forest training (50 trees, max depth 6)
5. Export decision tree as C if-else code

### Training Results

- Dataset: 2,696 samples (2,493 normal, 203 anomaly)
- Accuracy: 99.8%
- Feature importance: vib_z (51.3%) > vib_x (31.2%) > vib_y (8.9%) > humi (5.3%) > temp (3.4%)

### Deployment Strategy

After training, it was discovered that the single decision tree's vibration rules conflicted with tilt-only data (a tilted-but-stationary board also produces large single-axis values). The actual deployment uses a **dual-path detection** approach:

- **Vibration detection**: Sliding window variance analysis (5 samples, threshold 1,000,000)
- **Environmental detection**: Temperature/humidity rules extracted from the AI model

### Configuration

- Broker: Self-hosted MQTT server (edit `MQTT_BROKER_IP` in `src/mqtt_thread.c`)
- Port: 1883 (TCP)
- WebSocket: 9001 (for web dashboard)
- Topic: `edge-gateway/sensor`
- QoS: 1
- Publish interval: 5 seconds

## Web Dashboard

- Pure frontend HTML + JavaScript
- Connects to cloud server via MQTT WebSocket
- Displays real-time temperature, humidity, and 3-axis vibration
- Red flashing alert on anomaly detection
- Event log records the latest 100 entries

## Known Issues & Solutions

### 1. Sensor WHO_AM_I Returns 0x70 Instead of 0x68

The module actually uses an MPU6500 chip rather than MPU6050. They are pin-compatible with nearly identical registers, so no code changes were needed.

### 2. AI Model False Positives on Tilt

The training data did not include "tilted but stationary" samples, causing the model to misclassify large single-axis values from tilting as vibration anomalies.
**Solution**: Replaced tree-based vibration detection with sliding window variance analysis, which is tilt-invariant.

### 3. AI Status Overwritten by Sensor Thread

The `sensor_thread` overwrites the entire `g_shared.data` every 100ms, which wiped out the `ai_status` field written by the AI thread.
**Solution**: Moved `ai_status` from `sensor_data_t` to the outer `shared_data_t`, making it inaccessible to the sensor thread.

### 4. Web Dashboard WebSocket Connection Failure (HTTPS Restriction)

Browsers block insecure `ws://` connections when the page is loaded via HTTPS.
**Solution**: Enabled mosquitto WebSocket listener on port 9001 on the cloud server; access the dashboard via HTTP.

### 5. mosquitto_pub Missing Shared Library

After copying only the `mosquitto_pub` binary to the board, it reported `libmosquitto.so.1` missing.
**Solution**: Also copy the shared library file and run `ldconfig`.
